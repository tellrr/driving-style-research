## 4. Overall Strategic Position

We are **85–90% of the way to a fully evidence-based service blueprint**. Behavioral metrics, scoring architectures, and threshold ranges are mature enough to support production implementation with high confidence. The **single biggest blocker** is the absence of a unified, vehicle-category-specific threshold table with validated detection performance (sensitivity, specificity, false-positive rates) and contextual adjustment factors (weather, road surface, gradient) — without this, any deployed system will require extended field-calibration cycles and risk unacceptable false-alarm rates for non-passenger-car vehicle types.

## 6. What Would You Build?

### Ready Now

1. **Four-Metric Composite Driver Score (0–100, Deductive, with Probabilistic Interpretation)**
   The evidence is overwhelming and convergent. Use the AWS deductive model (start at 100, deduct per event) as the architectural skeleton, enriched with the SafeDriver-IQ probabilistic grounding so each score maps to an estimated crash probability. Implement the dual-tier severity taxonomy from the "Data Mining of Telematics Data" paper (0.5 G = harsh, 1.5 G = severe) for event classification. Weight the four behaviors using the risk-factor hierarchy confirmed across 35+ platforms. Deploy with Geotab's ~0.3 g baseline threshold for passenger vehicles, adjustable per fleet configuration. **This is ready for production today.**

2. **Adaptive Threshold Engine for Truck Fleets**
   The Jansen et al. truck threshold comparison study provides the direct empirical justification: a fixed threshold produces inconsistent detection across contexts. Combine this with the heavy-duty braking dynamics model (>5 m/s² service braking) and ARMOR's hybrid on-device/cloud processing pattern. Build a threshold engine that adjusts deceleration thresholds based on vehicle type (at minimum: passenger car vs. LCV vs. heavy truck) and speed context. **The engineering references are sufficient to ship a v1 with three vehicle categories.**

3. **Hybrid On-Device + Cloud Alert Pipeline with Coaching Integration**
   The AI-based coaching systematic mapping documents that hybrid architectures achieve up to 21% risky-behavior reduction. ARMOR's production documentation validates the on-device-for-immediacy / cloud-for-consistency design split. Build: on-device detection for immediate harsh-event alerts (audio/visual to driver), cloud-based aggregation for trend analysis and coaching report generation. Start with the four canonical event types plus fatigue proxy (steering entropy or continuous driving duration from tachograph data, per Geotab's fatigue monitoring framework).

### What to Defer

- **Per-event cost attribution models** (Cost & Maintenance pillar): We know the directional relationships (harsh driving → more wear → higher costs), but we lack the marginal-cost-per-event data required to build a credible "this harsh brake cost you €X" feature. Building this now would require guesswork. **Defer until per-event cost data is available.**
- **Road-surface and weather-adjusted thresholds**: The evidence clearly shows these adjustments are *needed* (Teletrac Navman, Jansen et al.), but no study provides calibrated adjustment factors. Shipping weather-adjusted thresholds now would be speculative. **Defer; plan for a field-calibration study using our own fleet data.**
- **Nationality/regional driving pattern normalization**: Cross-cultural differences are confirmed (Japanese/Chinese/Vietnamese study, German/Belgian i-DREAMS), but the evidence is too heterogeneous to derive scoring adjustments. **Defer; collect regional baseline data from early deployments.**

### Biggest Insight

**Driving styles are not static traits — they are dynamic, context-dependent states** ("Identification of dynamic driving styles based on behavioral primitives," [Nature Scientific Reports](https://www.nature.com/articles/s41598-026-38787-y)), and a single fixed threshold fails across contexts (Jansen et al. truck study). This finding should directly reshape the scoring architecture: rather than assigning a driver a persistent "aggressive" label, the system should model behavioral state transitions within and across trips, and thresholds must be parametric (varying by at minimum: vehicle type, speed band, and road context). Any design that treats a driver score as a fixed personality metric will be systematically inaccurate.
